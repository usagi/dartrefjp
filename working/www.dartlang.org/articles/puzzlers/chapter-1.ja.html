Dart ぱずらーず 第1章

著 Joshua Bloch 訳 Usagi Ito
2012年 1月（訳 2012年 3月）

もう何年も前に、私はJava ぱずらーず: ワナと落とし穴とコーナーケース（訳註: amazon、コーナーケース）という本を共同執筆した。 そして今、私は Dart チームに居て、その Java ぱずらーずの全てを Dart へと翻訳しようとしている。この試みがパズルを求める誰かさんや、Dart の設計に関する有用な知見としてお役に立てれば幸いである。 それでは、さっそく最初の章へと迷い込むとしよう。

ぱずらー 1: 奇数かな

次に示すJavaのメソッドでは引数が偶数か奇数かを表示するよ。でも、引数が負の奇数だと失敗しちゃうんだ。: 

public static boolean isOdd(int i) {
  return i % 2 == 1;
}

このパズルをDartで考えてみよう: Dartではこの様な関数もきっちりと奇数を判定してくれる（Javaとは違った挙動をするよ）: 

bool isOdd(int i) {
  return i % 2 == 1;
}

この問題を解決するこれ以上の答えは無いよね。でも、これは数学の「割り算の規則」を犯してしまっているんだ:

D = d * (D div d) + D mod d

（訳註: 数学的には割られる数が負であっても剰余は正）

C#、C++、C、Java、そしてJavaScript(訳註:こんな感じ)なんかとは互換性が無いね。この点についてはFlorian Loitsch(訳註:リンク先はdartlang.org本家)と議論しているよ。

ぱずらー 2: 変革の時

このプログラムは $2.00 から $0.90 支払った結果を計算して表示している。 

public class Change {
  public static void main(String args[]) {
    System.out.println(2.00 - 1.10);
  }
}

これを Dart に翻訳するとこうなる: 

main() {
  print(2.00 - 1.10);
}

Java（訳註: こんな感じ） でも Dartboard でも、これを実行すると 0.8999999999999999 と表示されてしまう。 Dart 仮想マシンを使うなら、この通貨表示は 0.9 となるのだけど、それは今のところはCの整形機能を通貨に適用しているからに過ぎないんだ（訳註: こんな感じ）。この点は将来、 Dart 仮想マシンでも Dartboard での実行結果と同様に表示されパズルの課題となるようにしないといけないね。

この問題に対応する為に、 Dart は10進浮動小数点演算をサポートすべきかな？ した方がいいって意見も沢山聞こえてきます。 例えば、 Mike Cowlishaw (IBM Fellow) はこうした問題に人生を掛けて取り組みました（http://speleotrove.com/decimal）。また、C#では 128-bit の10進浮動小数点をサポートする事でこの問題を避けました。でも、 JavaScript をデプロイしなければならない場合、10進浮動小数点の数値処理はとっても遅くて、何か他にもっと速くていい方法を使いたくなるでしょう(いやまあ、もしかしたら、IBMの Power6 チップなら前述の Mike Cowlishaw の尽力によってネイティブに10進浮動小数点をサポートしていたりもするんだけど)。ちなみに、 Cowlishaw が EcmaScript の委員だった頃、10進の算術は EcmaScript 4 の失敗を齎しました。

はてさて、10進浮動小数点の計算の為のライブラリを Dart が提供し、10進リテラルを言語サポートとすべきでしょうか？

実は、この浮動小数点と2進数の問題は JavaScript へコンパイルするよりも Dart の方が深刻だったりする。Javaならば、大抵のプログラマーはこの問題に取り組むとなれば double の使用を避け、 int を使えば良い事を知っています。でも、次の Dart のプログラムを考えてみましょう: 

main() {
  int i = 1000 * 1000 * 1000 * 1000 * 1000 * 1000 * 1000 * 1000;
  print(i == i + 1);
}

これを Dartboard で実行してみてご覧よ、 true って表示しちゃうから。とっても混乱しちゃったかい？ いやはや、私が思うにこれではこの問題には良い答えを出せないだろう。勿論、これが一時的な現状であって、やがては解決される問題だと思いたいけどね。

ぱずらー 3: 割り算の筆算

次の Java プログラムは マイクロ秒÷ミリ秒 の結果得られる1000までの数値を表示すると思うよね？でも、実際には 5 と表示されるよ（訳註: Java、C++）: 

public class LongDivision {
  public static void main(String[] args) {
    final long MICROS_PER_DAY = 24 * 60 * 60 * 1000 * 1000;
    final long MILLIS_PER_DAY = 24 * 60 * 60 * 1000;

    System.out.println(MICROS_PER_DAY / MILLIS_PER_DAY);
  }
}

このパズルはそのままじゃ Dart に持ち込めないんだ。 Dart は long 型を持っていないし、 int 型は任意精度だからね。もし、君が Dart で変数に int 型のラベルを付けてこんな計算をさせたとしても、答えは正しく 1000 と表示されるよ。 Dartboard でこの計算をすると正しい答えが得られるだろう。でも、一般に Dart の処理系が常に正しい答えを返してくれるとは限らないよ。

ぱずらー 4: このくらい基礎だよね 

public class Elementary {
  public static void main(String[] args) {
    System.out.println(12345 + 5432l); // Lowercase L masquerades as 1!
  }
}

このパズルもまた Dart には持ち込めない。 Dart には long (64-bit) リテラルが無いからね。

ぱずらー 5: 16進あそび

次の Java プログラムでは 1cafebabe が表示されると思うだろう？でも実際やってみると cafebabe と表示される（訳註: こんな感じ）。予想外の符号拡張（訳註: 符号拡張; sign extension）によってね(0xcafebabe は負の数だよ)。 

public class JoyOfHex {
  public static void main(String[] args) {
    System.out.println(
      Long.toHexString(0x100000000L + 0xcafebabe));
  }
}

この問題について Dart では任意精度の整数処理をしているから心配要らないよ。次の Dart プログラムは 1cafebabe を表示するよ: 

main() {
  print((0x100000000 + 0xcafebabe).toRadixString(16));
}

ぱずらー 6: マルチキャスト 

public class Multicast {
  public static void main(String[] args) {
    System.out.println((int) (char) (byte) -1);
  }
}

This puzzle doesn't apply in Dart as it has no casts.

Puzzler 7: Swap Meat

This program attempts to use a hideous hack to swap the contents of two int variables: 

public class CleverSwap {
  public static void main(String[] args) {
    int x = 1984;
    int y = 2001;
    x ^= y ^= x ^= y;
    System.out.println("x = " + x + "; y = " + y);
  }
}

In Dart: 

main() {
  int x = 1984;
  int y = 2001;
  x ^= y ^= x ^= y;
  print("x = ${x}; y = ${y}");
}

This puzzler behaves exactly the same in Java and Dart. Luckily, it doesn't indicate a language problem: It's just a cautionary tale regarding overly clever programming and premature optimization. (For what it’s worth, the Rube Goldberg “fix” in the book also applies equally well to Dart and Java.)

Puzzler 8: Dos Equis

This program would appear to print XX, but actually prints X88 in Java: 

public class DosEquis {
  public static void main(String[] args) {
    char x = 'X';
    int i = 0;
    System.out.print(true  ? x : 0);
    System.out.print(false ? i : x);
  }
}

In Dart: 

main() {
  string x = 'X';
  int i = 0;
  print(true  ? x : 0);
  print(false ? i : x);
}

This puzzle is fixed in Dart, because Dart doesn’t have a char type that is a (barely) glorified int type, doesn’t do primitive widening conversions on ternary expressions, and doesn't have weird rules concerning small int values (don't ask; read the book if you’re morbidly curious).

Puzzler 9: Tweedledum

In this puzzle, you’re asked to provide declarations to cause the indicated behavior: 

Public class Tweedledum {
  public static void main(String[] args) {
    // Put your declarations for x and i here

    x += i;     // Must be LEGAL
    x = x + i;  // Must be ILLEGAL
  }
}

Fixed in Dart: no short type, no implicit narrowing conversion on compound assignment expressions.

Puzzler 10: Tweedledee

Again, you’re asked to provide declarations to cause the indicated behavior: 

public class Tweedledee {
  public static void main(String[] args) {
    // Put your declarations for x and i here

    x = x + i;  // Must be LEGAL
    x += i;     // Must be ILLEGAL
  }
}

This one is fixed in Dart; in fact, it’s even fixed in Java, as of 1.6. It had to do with peculiarities of Java's String concatenation operator and its corresponding compound assignment operator.

Summary

Out of ten Java puzzlers, eight have been fixed in Dart, one (Puzzler 3) arguably made worse (it's easier to get into floating point binary troubles), and one doesn't really indicate a language problem (Puzzler 7).
